# HeyDoc Trunk Constraints Reference

**Document ID:** `heydoc-grounding:trunk-constraints:2026-06`  
**Version:** 1.0.0  
**Last reviewed:** 2026-06-23  
**Citation ID:** `trunk-constraints:v1.0.0:2026-06`

This document is the canonical reference for all nine trunk agents' roles, clinical constraints, output contracts, and hard stops. It is retrieved by the docs MCP server when trunks need to ground scope assertions, and it is the authoritative source used by `integration/trunk-pipeline.js` `TRUNK_CONSTRAINTS` to inject `ContextPacket.constraints[]`.

---

## Universal Constraints (all trunks, non-negotiable)

These constraints apply to every trunk regardless of specialisation. They are injected into every `ContextPacket.constraints[]` array before trunk-specific constraints are appended.

| Constraint | Rationale |
|---|---|
| **No diagnosis** | All diagnostic output is provisional, requires clinician confirmation, and must never be framed as definitive via telehealth chat. |
| **No dosages** | Specific dose values may only appear as output of `mcp-pharmacology` pharm.check — never generated by the LLM from parametric memory. |
| **No invented codes** | SNOMED CT, ICD-10-AM, LOINC, PBS codes must be backed by a `mcp-terminology` lookup receipt in the ContextPacket. |
| **No invented operational facts** | IHI numbers, lab results, pharmacy availability, delivery status, ECG readings must come from live-data receipts. |
| **No invented service names** | Internal component names not in the gap register Allowed Service Registry must not appear in output. |
| **HARD_FAIL blocks** | A `HARD_FAIL` from `mcp-pharmacology` pharm.check must block pipeline continuation — no trunk may override or ignore it. |
| **Australia jurisdiction** | All clinical guideline references must be AU-aligned. Use AU sources present in context (eTG, Choosing Wisely AU, NHFA/CSANZ, RACGP) only when citation IDs are provided in the ContextPacket. |

## Trunk-by-Trunk Reference

### Trunk 1.0 — Master Intake and Safety Gate

**Role:** First-pass intake normalisation and safety gate. Routes the session to downstream trunks. Identifies immediate safety red flags requiring escalation before any other trunk work.

**Output contract:**
1. `intake_summary` — concise normalised summary of known facts
2. `safety_gate` — `status`: `clear` | `escalate_now` | `blocked_incomplete`; `reasons`: list
3. `routing_plan` — `next_trunks`: ordered list; `why`: short rationale
4. `missing_inputs` — unanswered questions or missing receipts blocking safe progression
5. `evidence_refs` — citation/receipt refs for non-obvious claims

**Trunk-specific constraints:**
- Initial routing and safety gate only — no history enrichment
- `escalate_now` must be returned immediately if any T5 red flag is detected, without waiting for downstream trunks

**Hard stops:**
- Must not produce `routing_plan.next_trunks` without completing `safety_gate` first
- `blocked_incomplete` status means no routing plan is produced — list missing inputs and stop

**Verifier checks triggered:** `no_invented_codes`, `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`, `hard_stop_enforcement`

---

### Trunk 2.0 — Clinical Triage

**Role:** Triage-only agent. Gathers history, screens for red flags, produces routing/escalation suggestions. Does not enriched history or produce problem representations — those belong to Trunks 3.0 and 4.0.

**Output contract:**
- Follow-up questions for red-flag screening
- Triage routing payload (when to escalate, when to continue)
- Citation IDs from context cited inline (e.g., `per citation cw-au:…`) so verifier can match claims to proof artifacts

**Trunk-specific constraints:**
- Triage protocol only — no history structuring or problem representation
- Guideline claims (e.g., "Choosing Wisely recommends…") must include the `citation_id` from the ContextPacket evidence — not recited from memory

**Hard stops:**
- Any output asserting a specific guideline without a citation ID in evidence is a verifier `no_invented_guidelines` failure

**Verifier checks triggered:** `no_invented_codes`, `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`

---

### Trunk 3.0 — Structured History Enrichment

**Role:** Expands patient narrative into structured triage history for downstream trunks. Produces targeted follow-up questions, missing-information flags, and a structured summary payload. Does not produce problem representation or risk framing — Trunk 4.0 does that.

**Output contract:**
1. `follow_up_questions` — minimal list of questions to complete triage history; no duplicates from prior turns
2. `structured_history` — key-value structure with known facts and explicit `unknown` markers
3. `evidence_refs` — citation/receipt references for non-obvious claims

**Trunk-specific constraints:**
- History enrichment only — symptom characterisation, onset/progression, relevant negatives, red-flag screening prompts, risk-context clarification
- Unknowns must be labelled `unknown` — not inferred or fabricated
- Questions must be clinically focused and minimal; avoid repeating questions already answered in facts

**Hard stops:**
- Must not produce a diagnosis or differential, even implicitly via question framing (e.g., "Do you have GORD?" reveals the differential — use "Do you have any known stomach or oesophageal conditions?" instead)

**Verifier checks triggered:** `no_invented_codes`, `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`

---

### Trunk 4.0 — Problem Representation and Risk Framing

**Role:** Converts the structured history from Trunk 3.0 into a concise, neutral problem representation for decision trunks (5.0–9.0). Separates immediate concerns from routine follow-up needs. Flags data quality limits.

**Output contract:**
1. `problem_representation` — one concise paragraph of grounded facts only
2. `risk_frame`:
   - `immediate_concerns` — short bullet list
   - `routine_follow_up` — short bullet list
3. `data_gaps` — explicit unknowns or missing receipts/citations
4. `evidence_refs` — citations/receipts used for non-obvious claims

**Trunk-specific constraints:**
- Problem representation and risk framing only — no differential diagnosis, no management plan
- Must separate confirmed facts from inferred conclusions — problem representation contains only confirmed facts
- Data gaps must be listed explicitly, not glossed over

**Hard stops:**
- A problem representation that contains inferential language ("likely", "probably", "consistent with") without a citation to back it is a verifier failure — use "unknown" or "unclear" for unconfirmed states

**Verifier checks triggered:** `no_invented_codes`, `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`

---

### Trunk 5.0 — Axis B Deterministic Rule-Out

**Role:** Translates the problem representation into an Axis B rule-out checklist — a deterministic matrix of required negatives and confirmations that must be established before the AI Doctor can proceed toward management. Surfaces blocking gaps.

**Output contract:**
1. `axis_b_ruleout_matrix`:
   - `required_negatives` — list with status `confirmed` | `unknown`
   - `required_confirmations` — list with status
   - `required_evidence` — citation/receipt requirements per item
2. `blocking_gaps` — items requiring resolution before progression
3. `next_data_requests` — minimal follow-up questions or retrieval needs
4. `evidence_refs` — citation/receipt references used

**Trunk-specific constraints:**
- Axis B rule-out per approved templates only — templates must come from the knowledge MCP (structured KG dataset `axis-b-templates`), not from LLM memory
- `unknown` must be used for any item without proof — never infer a negative
- A "required negative" that is `unknown` is a blocking gap, not a safe assumption

**Hard stops:**
- If the `axis-b-templates` knowledge dataset is absent from the ContextPacket, output must be `blocked_no_templates` — do not construct rule-out matrices from memory
- Any required negative that the AI Doctor has not explicitly confirmed must be flagged, even if clinically unlikely

**Verifier checks triggered:** `no_invented_codes`, `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`

---

### Trunk 6.0 — Investigation Interpretation

**Role:** Interprets already-provided, sanitised investigation summaries from the ContextPacket. Classifies findings by urgency. Identifies which results require immediate escalation versus routine follow-up. Does not order or request investigations — that belongs to management planning.

**Output contract:**
1. `finding_summary`:
   - `critical` — list
   - `abnormal_noncritical` — list
   - `normal_or_expected` — list
   - `insufficient_data` — list (for results not in ContextPacket)
2. `escalation_signal`:
   - `requires_urgent_escalation` — `true` | `false`
   - `reason` — short grounded explanation
3. `next_data_requests` — minimal missing items
4. `evidence_refs` — citation/receipt refs

**Trunk-specific constraints:**
- Investigation interpretation only — no investigation ordering, no management plan
- LOINC-derived evidence required for coded investigation assertions — any reference to a lab code must come from a `mcp-terminology` LOINC receipt in the ContextPacket
- Raw numeric lab values must not be reproduced in output — only sanitised interpretation (e.g., "Troponin I elevated — critical threshold exceeded", not "250 ng/L")
- `insufficient_data` must be used for any result not present in the ContextPacket

**Hard stops:**
- Must not reproduce raw numeric investigation values — they must be sanitised before entering the ContextPacket (sanitised_by field in context-packet.schema.json), and Trunk 6.0 must work only with the sanitised form
- An investigation result not in the ContextPacket must not be fabricated or inferred

**Verifier checks triggered:** `no_invented_codes`, `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`

### Trunk 7.0 — Code Lock-In

**Role:** Converts grounded clinical concepts into stable coded outputs for downstream systems — SNOMED CT and ICD-10-AM. Applies benign-registry gating. The only trunk that produces coded clinical identifiers; all other trunks must not emit codes without Trunk 7.0 having locked them.

**Output contract:**
1. `candidate_codes` — list of SNOMED/ICD concepts with evidence refs for each
2. `code_lock_status` — `locked` | `blocked`
3. `blocking_reasons` — reasons when blocked (missing receipt, ambiguous mapping, policy gate)
4. `benign_registry_gate` — status and rationale when applicable
5. `evidence_refs` — terminology receipt IDs and citations used

**Trunk-specific constraints:**
- Code lock-in requires terminology receipt — every code emitted must map to a `mcp-terminology` lookup receipt in the ContextPacket
- Benign registry gating applies when the benign registry knowledge dataset is present in the ContextPacket — Trunk 7.0 must not bypass it
- `blocked` status with explicit reasons is preferable to emitting codes without receipts

**Hard stops:**
- Emitting a SNOMED CT or ICD-10-AM code without a matching terminology lookup receipt in the ContextPacket is a `no_invented_codes` verifier failure with severity=critical
- If the terminology server receipt is absent, `code_lock_status: blocked` must be returned — never emit codes from parametric memory

**Verifier checks triggered:** `no_invented_codes` (critical), `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`

### Trunk 8.0 — Pharmacology Firewall Intent Check

**Role:** Converts clinical intent into a structured pharmacology safety check request (`pharm_intent_payload`). Gates pipeline continuation based on deterministic `mcp-pharmacology` firewall outcomes. The only trunk that interacts with the pharmacology MCP server.

**Output contract:**
1. `pharm_intent_payload` — structured PharmIntent object (see `mcp/schemas/pharm-intent.schema.json`)
2. `firewall_status` — `PASS` | `WARN` | `HARD_FAIL` | `BLOCKED_NO_PROOF`
3. `blocking_reasons` — required for `HARD_FAIL` or `BLOCKED_NO_PROOF`
4. `next_data_requests` — minimal facts needed for safe pharmacology evaluation
5. `evidence_refs` — receipts/citations used

**Trunk-specific constraints:**
- No dosages — pharm_intent_payload must not contain specific dose values; only drug identity, class, and route are permitted (doses are output of pharm.check, not input from the LLM)
- No autonomous prescribing — pharmacology firewall HARD_FAIL blocks continuation unconditionally
- Pharmacology claims must be backed by `mcp-pharmacology` receipts — allergy status, interactions, renal adjustments must not be invented from parametric memory

**Hard stops:**
- A `HARD_FAIL` from pharm.check must be surfaced as `firewall_status: HARD_FAIL` and must block any management recommendation. There is no override.
- `BLOCKED_NO_PROOF` must be returned when required patient facts (allergy status, current medications, renal function) are absent from the ContextPacket
- Schedule 8 (S8) intent without a documented PDMP (SafeScript WA) check in the ContextPacket must return `HARD_FAIL`

**Verifier checks triggered:** `no_invented_codes`, `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`, `hard_stop_enforcement` (critical)

---

### Trunk 9.0 — Red-Flag Questionnaire and Escalation Gate

**Role:** Produces a deterministic red-flag questionnaire flow keyed to the SNOMED-coded condition in the ContextPacket. Classifies outcomes into escalation tiers. Final safety gate before management output.

**Output contract:**
1. `red_flag_questionnaire`:
   - `questions` — list of concise red-flag questions
   - `responses` — known responses or `unknown`
2. `risk_outcome` — `escalate_now` | `urgent_review` | `routine_follow_up` | `blocked_incomplete`
3. `blocking_items` — unanswered questions or missing receipts preventing safe completion
4. `next_actions` — minimal next steps consistent with risk outcome (non-diagnostic, non-dosage)
5. `evidence_refs` — citation/receipt refs used

**Trunk-specific constraints:**
- Red-flag questionnaire driven by grounded evidence only — questions must correspond to evidence in the ContextPacket, not free-form LLM generation
- Red-flag items must be keyed by SNOMED code from a `mcp-terminology` receipt — not inferred from the patient's narrative
- `blocked_incomplete` must be returned when required questionnaire items cannot be answered from available facts
- `escalate_now` must be returned immediately when any answer triggers a T5 safety-netting condition — no deferral

**Hard stops:**
- An `escalate_now` output must not be accompanied by routine management recommendations — escalation and concurrent routine advice creates ambiguity that could delay emergency response
- The questionnaire bank must come from the `redflags-*` knowledge dataset in the ContextPacket — if absent, `blocked_no_questionnaire_data` must be returned

**Verifier checks triggered:** `no_invented_codes` (critical for SNOMED keying), `no_invented_guidelines`, `no_invented_operations`, `no_repo_invention`, `hard_stop_enforcement`

## Constraint-to-Verifier Check Mapping

| Constraint | Verifier Check | Severity |
|---|---|---|
| No invented SNOMED/ICD code | `no_invented_codes` | critical |
| No invented guideline claim | `no_invented_guidelines` | fail |
| No invented IHI/lab/pharmacy | `no_invented_operations` | critical |
| No invented service name | `no_repo_invention` | warning |
| HARD_FAIL must block | `hard_stop_enforcement` | critical |

## Trunk Routing Logic

Trunk 1.0 selects downstream trunks in `routing_plan.next_trunks`. Standard routing sequences:

| Clinical context | Trunk sequence |
|---|---|
| Acute / unclear presentation | 1.0 → 2.0 → 3.0 → 4.0 → 9.0 |
| Structured history available | 1.0 → 4.0 → 5.0 → 9.0 |
| Investigation results present | 1.0 → 6.0 → 4.0 → 9.0 |
| Code lock-in required | ... → 7.0 |
| Pharmacology question present | ... → 8.0 |
| Red-flag escalation needed | 1.0 → 9.0 (short-circuit) |

Trunks may be parallelised when their inputs are independent. Trunk 8.0 must always run before any management recommendation that involves medication. Trunk 9.0 is always the final gate before escalation output.
